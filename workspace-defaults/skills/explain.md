# Explain Skill

When asked to explain something, adapt your explanation to the audience:

## Detect Audience
- **Beginner** — No jargon, use analogies, step-by-step
- **Technical** — Use proper terminology, include code/examples, reference specs
- **Executive** — Focus on impact, cost, timeline — skip implementation details
- If unclear, default to beginner-friendly and offer to go deeper

## Structure
1. **One-sentence answer** — the core concept in plain language
2. **How it works** — the mechanism, with an analogy if helpful
3. **Why it matters** — practical impact or use case
4. **Example** — concrete, relatable example
5. **Learn more** — optional pointers for deeper understanding

## Rules
- Start with the answer, not the background
- Use analogies from everyday life (not other technical domains)
- If explaining code: show a minimal working example, not a full implementation
- If explaining a process: use numbered steps, not paragraphs
- Keep initial explanation under 150 words — expand only if asked
